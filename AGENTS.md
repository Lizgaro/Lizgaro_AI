# AGENTS.md — Portfolio Site

## Роль

Ты работаешь внутри отдельного проекта `portfolio-site`: русскоязычный сайт-портфолио Артёма Лизгаро на Next.js.

Цель проекта: публичный актив, который объясняет, кто такой Артём, показывает услуги/кейсы/мышление и приводит к заявкам.

## Граница контекста

По умолчанию работай только с этой папкой:

- `README.md`
- `app/`
- `components/`
- `content/blog/`
- `data/`
- `lib/`
- `public/`

Не читай весь `C:\LA\life-os-v0.3` без причины.

К корневому Life OS обращайся только если задача требует личного контекста, позиционирования или проектной стратегии. Тогда читать минимально:

1. `..\PERSONAL_AI_CONTEXT.md`
2. `..\projects\portfolio-site.md`
3. `..\projects\portfolio-prd.md`
4. `..\systems\codex-portfolio-build-brief.md`
5. `..\references\portfolio-references.md`

## Приоритеты внутри проекта

1. Сайт должен быть понятен и готов к отправке потенциальному клиенту.
2. CTA, офферы, кейсы и доверие важнее декоративных улучшений.
3. Блог нужен для SEO/GEO и публичного пути, но не должен ломать главный лендинг.
4. Не добавлять CMS, сложные анимации и лишнюю архитектуру до стабильного контент-процесса.

### Короткий регламент публикации статьи

Если Артём пишет “опубликуй статью”, “залей статью в блог”, “пушни статью на сайт”:

1. Не читай весь Life OS и не ищи новый пайплайн. Достаточно:
   - `AGENTS.md`;
   - `README.md`;
   - `content/blog/_draft-example.md`;
   - 1-2 опубликованные статьи в `content/blog/` как стиль.
2. Создай или обнови один markdown-файл в `content/blog/`.
3. Адаптируй текст под текущий стиль сайта, а не вставляй сырой бриф:
   - сильный первый абзац;
   - `## Суть`;
   - короткие секции;
   - маркеры `**Коротко:**` и `**Маркер:**`;
   - списки и цитаты;
   - FAQ, если он есть в исходнике;
   - один мягкий CTA в конце.
4. Поставь `status: "published"` только если задача именно про публикацию.
5. Проверь:

```powershell
npm run lint
npm run build
```

6. В `git add` добавляй только нужную статью и реально изменённые инструкции/код. Не добавляй случайные черновики.
7. Коммит:

```powershell
git commit -m "feat: publish [short article topic] article"
git push origin main
```

8. Если `git push` падает из-за DNS к `github.com`, не меняй код и не делай новый коммит. Сначала повтори push. Если снова падает, попробуй временный Git-обход:

```powershell
git -c http.sslBackend=schannel -c http.version=HTTP/1.1 push origin main
```

Если и это не помогло — сообщи, что commit готов локально, но push заблокирован DNS/сетью.

9. Если в проекте подключён Vercel Git integration, обычный `git push origin main` запускает деплой. Не трать время на Vercel CLI, если пользователь не просил отдельный ручной деплой.

1. Не придумывай новый процесс. Всегда работай через `content/blog/`.
2. Если статья новая — создай файл `content/blog/slug-latinicey.md`.
3. Используй стиль двух текущих опубликованных статей: сильный лид, короткие секции, маркеры, списки, цитаты, один понятный CTA.
4. Базовая структура:

```txt
лид -> суть -> где теряется человек -> сценарий решения -> связь с подходом -> действие -> вывод
```

5. Используй визуальные маркеры, которые уже поддерживает рендерер:

```md
**Коротко:** главный тезис.
**Маркер:** симптом, правило или быстрый вывод.
> Главная мысль, которую нужно запомнить.
```

6. Не делай статью простыней. После 2-4 обычных абзацев должен быть маркер, список, цитата, подзаголовок или пошаговый блок.
7. Для черновика ставь `status: "draft"`. Для публикации после проверки ставь `status: "published"`.
8. Перед push обязательно запускай:

```powershell
npm run lint
npm run build
```

9. Не запускай `npm run build`, пока работает `npm run dev`. Сначала останови dev-сервер через `Ctrl + C`, иначе Next.js может сломать `.next` кэш.
10. После успешной проверки пушь:

```powershell
git add content/blog/имя-статьи.md README.md AGENTS.md app/globals.css lib/blog.ts
git commit -m "feat: publish blog article"
git push origin main
```

Добавляй в `git add` только реально изменённые файлы. Не добавляй чужие черновики и случайные untracked-файлы.

Статья должна иметь frontmatter:

```md
---
title: ""
description: ""
slug: ""
date: "YYYY-MM-DD"
category: ""
tags:
  - ""
author: "Артём Лизгаро"
status: "draft"
sourceType: "manual"
publishToSocial: false
---
```

`status: "draft"` — не публикуется.

`status: "published"` — попадает в публичный блог, sitemap/rss/страницы тегов и категорий.

Не ставь `published`, если Артём попросил только черновик или SEO-подготовку. Если задача звучит “подготовь к публикации” или “опубликуй в блог”, можно ставить `published` после проверки.

## SEO/GEO-подготовка

Если статья приходит из отдельного SEO-процесса, источник обычно здесь:

```txt
..\departments\content-seo\ready-for-blog\
```

Перед добавлением в блог проверь:

- заголовок понятен без контекста;
- description коротко продает смысл;
- slug латиницей;
- есть категория и 3-6 тегов;
- нет выдуманных метрик, отзывов и кейсов;
- есть связь с услугами, проектами или личным путем Артёма;
- CTA не выглядит навязчиво.

## Команды

Установить зависимости:

```powershell
npm install
```

Запустить локально:

```powershell
npm run dev
```

Проверить перед завершением:

```powershell
npm run lint
npm run build
```

Открыть сайт:

```txt
http://localhost:3000
```

Команды подходят для PowerShell, Command Prompt и Warp. Главное — сначала перейти в папку:

```powershell
cd C:\LA\life-os-v0.3\portfolio-site
```
